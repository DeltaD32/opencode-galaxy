# Projects DB — opencode.db Extension Design

> **Status:** Phases 3–4.6 complete  
> **Last updated:** 2026-06-26  
> **Depends on:** Blackboard Architecture (Phase 2), see `ARCHITECTURE.md`

---

## Decision Summary

Extend `opencode.db` with new project-tracking tables rather than creating a
separate database or mirroring to the memory MCP.

**Rejected alternatives:**

| Option | Why rejected |
|---|---|
| Mirror to memory MCP | Two sources of truth; sync drift risk; memory MCP schema is flat (entities + relations only, no rich fields) |
| Separate `projects.db` + local REST API | Extra process to manage; separate file; operational overhead; galaxy needs a merge layer |
| **opencode.db extension (chosen)** | Single source of truth; no sync; galaxy already has access to this file; secretary and galaxy both in same process context |

---

## Database Location

```
~/.local/share/opencode/opencode.db
```

The existing file used by OpenCode for sessions, messages, and routing cache pairs.
New tables are additive — existing tables are untouched.

## Schema

> **Source of truth:** `~/.opencode/skills/projects/projects.py` — `_SCHEMA_SQL` constant  
> plus idempotent column additions in `_apply_phase4_migrations()` and `_apply_phase46_migrations()`.  
> The SQL below is exact (copy-verified 2026-06-26).

```sql
-- A named project grouping related blackboard tasks
CREATE TABLE IF NOT EXISTS projects (
    id          TEXT PRIMARY KEY,           -- UUID4 generated by _new_id()
    name        TEXT NOT NULL,
    description TEXT,
    status      TEXT NOT NULL DEFAULT 'active',
    --   valid values: 'active' | 'paused' | 'complete' | 'archived'
    --   (enforced by PROJECT_STATUSES frozenset in projects.py)
    created_at  TEXT NOT NULL,              -- ISO 8601 UTC e.g. 2026-06-26T08:00:00+00:00
    updated_at  TEXT NOT NULL
    -- Phase 4.6 columns (added via ALTER TABLE ADD COLUMN — idempotent):
    -- distillation_ready  INTEGER NOT NULL DEFAULT 0
    --   1 = project marked complete, awaiting user-prompted distillation
    -- distilled_at        TEXT
    --   ISO 8601 timestamp set when distil_project() finishes; clears distillation_ready
);

-- A single task blackboard within a project (or standalone)
CREATE TABLE IF NOT EXISTS blackboards (
    id               TEXT PRIMARY KEY,      -- UUID4
    project_id       TEXT REFERENCES projects(id),
    --   NULL → standalone task; non-null → linked to a project
    task_description TEXT NOT NULL,
    status           TEXT NOT NULL DEFAULT 'deliberating',
    --   valid values: 'deliberating' | 'awaiting-approval' | 'executing' | 'done' | 'blocked'
    --   (enforced by BLACKBOARD_STATUSES frozenset in projects.py
    --    and VALID_STATUSES frozenset in blackboard.py)
    --   transitions:
    --     deliberating  →  awaiting-approval  (request_approval() called)
    --     awaiting-approval  →  executing     (user approves; mark_status('executing'))
    --     deliberating       →  executing     (auto-proceed, low-risk; mark_status('executing'))
    --     executing          →  done          (worker completes; mark_status('done'))
    --     executing          →  blocked       (worker fails; mark_status('blocked'))
    --     deliberating       →  blocked       (dependency unresolved; mark_status('blocked'))
    file_path        TEXT NOT NULL,         -- /tmp/opencode/task-YYYYMMDD-NNN.md
    created_at       TEXT NOT NULL,
    updated_at       TEXT NOT NULL
    -- Phase 4 columns (added via ALTER TABLE ADD COLUMN — idempotent):
    -- depends_on       TEXT
    --   JSON array of blackboard UUIDs that must reach status='done' before this one executes
    --   e.g. '["uuid-a","uuid-b"]' or NULL (no dependencies)
    -- approval_required INTEGER NOT NULL DEFAULT 0
    --   1 = worker is blocked until approve_blackboard() is called; 0 = auto-proceed
    -- approved_by      TEXT
    --   'user' (explicit user confirmation) | 'auto' (low-risk auto-proceed) | NULL (not yet approved)
    -- approved_at      TEXT
    --   ISO 8601 timestamp set by approve_blackboard()
);

-- Individual specialist sections written to a blackboard
--
-- *** ONE ROW PER SECTION NAME, NOT PER SPECIALIST TURN ***
--
-- append_section() in blackboard.py is idempotent:
--   if (blackboard_id, section_name) already exists → DELETE old row, INSERT new row
--   (upsert via explicit DELETE + INSERT, not SQL UPSERT/ON CONFLICT)
-- sync_blackboard_sections() in projects.py uses the same pattern.
--
-- Consequence: if programming-expert refines its "Programming Analysis" section,
-- the previous row is replaced — you only ever have the latest version.
-- There is NO append-only log of specialist edits; the section table holds
-- current state only.
--
-- Sections that survive Phase 4.6 compression (compressed stays 0):
--   'Execution Plan', 'Execution Result'
-- Sections that are soft-deleted on blackboard done (compressed set to 1):
--   all other section names (raw specialist analysis)
CREATE TABLE IF NOT EXISTS sections (
    id            TEXT PRIMARY KEY,         -- UUID4
    blackboard_id TEXT NOT NULL REFERENCES blackboards(id),
    agent         TEXT NOT NULL,            -- e.g. 'programming-expert', 'request-orchestrator'
    section_name  TEXT NOT NULL,            -- e.g. 'Programming Analysis', 'Execution Plan'
    content       TEXT NOT NULL,            -- full markdown body (metadata lines stripped)
    written_at    TEXT NOT NULL             -- ISO 8601, taken from **Written:** line or _now_iso()
    -- Phase 4.6 column (added via ALTER TABLE ADD COLUMN — idempotent):
    -- compressed    INTEGER NOT NULL DEFAULT 0
    --   0 = visible in normal queries
    --   1 = soft-deleted (raw specialist analysis hidden; content preserved in DB)
);

-- Architectural / technical decisions made during a blackboard task
CREATE TABLE IF NOT EXISTS decisions (
    id            TEXT PRIMARY KEY,         -- UUID4
    blackboard_id TEXT NOT NULL REFERENCES blackboards(id),
    made_by       TEXT NOT NULL,            -- agent name or 'user'
    decision      TEXT NOT NULL,            -- short label e.g. 'use signals not NgRx'
    rationale     TEXT,                     -- why this was chosen (optional)
    timestamp     TEXT NOT NULL,            -- ISO 8601
    influenced    TEXT                      -- JSON array of future blackboard UUIDs, or NULL
);

-- Conflicts between specialist recommendations on a blackboard
CREATE TABLE IF NOT EXISTS conflicts (
    id            TEXT PRIMARY KEY,         -- UUID4
    blackboard_id TEXT NOT NULL REFERENCES blackboards(id),
    agent_a       TEXT NOT NULL,
    agent_b       TEXT NOT NULL,
    description   TEXT NOT NULL,            -- what they disagreed on
    resolved      INTEGER NOT NULL DEFAULT 0,  -- 0 = open, 1 = resolved
    resolution    TEXT,                     -- how it was resolved (may include [enforced-decision:uuid])
    resolved_by   TEXT,                     -- 'secretary' | 'user'
    resolved_at   TEXT                      -- ISO 8601
);

-- Sequential specialist handoff queue (one row per specialist per blackboard)
-- Fan-out is sequential because the task tool cannot spawn agents in parallel.
CREATE TABLE IF NOT EXISTS specialist_queue (
    id            TEXT PRIMARY KEY,         -- UUID4
    blackboard_id TEXT NOT NULL REFERENCES blackboards(id),
    agent         TEXT NOT NULL,
    status        TEXT NOT NULL DEFAULT 'pending',
    --   valid values: 'pending' | 'active' | 'done' | 'skipped'
    --   (enforced by QUEUE_STATUSES frozenset in projects.py)
    queued_at     TEXT NOT NULL,            -- ISO 8601
    started_at    TEXT,                     -- set when status → 'active'
    completed_at  TEXT                      -- set when status → 'done'
);

-- Indexes for common access patterns
CREATE INDEX IF NOT EXISTS idx_blackboards_project   ON blackboards(project_id);
CREATE INDEX IF NOT EXISTS idx_blackboards_status    ON blackboards(status);
CREATE INDEX IF NOT EXISTS idx_sections_blackboard   ON sections(blackboard_id);
CREATE INDEX IF NOT EXISTS idx_decisions_blackboard  ON decisions(blackboard_id);
CREATE INDEX IF NOT EXISTS idx_conflicts_blackboard  ON conflicts(blackboard_id);
CREATE INDEX IF NOT EXISTS idx_conflicts_resolved    ON conflicts(resolved);
CREATE INDEX IF NOT EXISTS idx_queue_blackboard      ON specialist_queue(blackboard_id);
CREATE INDEX IF NOT EXISTS idx_queue_status          ON specialist_queue(status);
```

## Status Strings — Exact Values

### `blackboards.status` / `projects.status`

All status strings are enforced via `frozenset` constants in `projects.py` and `blackboard.py`.
Passing any other string raises `ValueError`.

#### Blackboard statuses (`BLACKBOARD_STATUSES` / `VALID_STATUSES`)

| Status | Meaning | How set |
|---|---|---|
| `deliberating` | Default — specialists are writing sections | `create()` initial value; `mark_status()` |
| `awaiting-approval` | Execution Plan written, waiting for user OK | `request_approval()` calls `mark_status('awaiting-approval')` |
| `executing` | Approved — worker is running | `mark_status('executing')` + `approve_blackboard()` |
| `done` | Worker finished successfully | `mark_status('done')` — triggers `compress_sections()` + `auto_archive_if_done()` |
| `blocked` | Worker failed or dependency unresolved | `mark_status('blocked')` |

> **`request_approval()`** always sets status to **`awaiting-approval`** (not `blocked`,
> not `executing`). The gate is then opened only after `mark_status('executing')` is called
> following user confirmation.

> **`is_gate_open()`** in `blackboard.py` returns `True` only when status is exactly
> `'executing'`. The worker checks this as a hard gate before doing any work.

#### Project statuses (`PROJECT_STATUSES`)

| Status | Meaning |
|---|---|
| `active` | Work is ongoing |
| `paused` | Temporarily on hold |
| `complete` | All work done — triggers `mark_distillation_ready()` automatically |
| `archived` | Historical record only |

## `sections` Table — Row Semantics

**One row per section name, not per specialist turn.**

`append_section()` in `blackboard.py` is explicitly documented as idempotent:

```python
# From blackboard.py append_section():
if pattern.search(existing):
    # Replace existing section — DELETE old row, INSERT new row in sync_blackboard_sections()
    updated = pattern.sub(new_block.lstrip("\n"), existing)
else:
    # Append new section
    updated = existing.rstrip("\n") + "\n" + new_block
```

`sync_blackboard_sections()` in `projects.py` mirrors this:

```python
# DELETE existing row for (blackboard_id, section_name) first, then INSERT fresh
conn.execute(
    "DELETE FROM sections WHERE blackboard_id = ? AND section_name = ?",
    (blackboard_id, section_name),
)
conn.execute(
    "INSERT INTO sections (id, blackboard_id, agent, section_name, content, written_at) ...",
    ...
)
```

**Practical consequences:**

| Scenario | Result |
|---|---|
| `programming-expert` writes `## Programming Analysis` once | 1 row: `agent='programming-expert'`, `section_name='Programming Analysis'` |
| `programming-expert` re-writes the same section | Still 1 row — old row deleted, new row inserted with fresh `written_at` |
| Two specialists write sections with different names | 2 rows — one per name |
| Two specialists both write `## Proposed Changes` | 1 row — the later call wins |
| A specialist writes the section, then a second specialist writes the same section name | 1 row — second specialist's content replaces first; `agent` column reflects second writer |

**There is no edit history in the sections table.** If you need to audit who wrote what and when,
read the blackboard `.md` file directly — `append_section()` always stamps a `**Written:**` line
in the file body, but the DB only tracks the latest version per section name.

## Access Pattern

### Secretary (write + read)

The secretary is the **only agent** that writes to these tables. It uses `sqlite3`
via the clipjoint venv `bash` tool — restricted to DB operations and blackboard
file reads only.

```bash
# Secretary pattern — create a blackboard record
~/.opencode/plugins/clipjoint/.venv/bin/python3 - << 'EOF'
import sqlite3, json, uuid
from datetime import datetime, timezone

db = sqlite3.connect("/Users/QTE2362/.local/share/opencode/opencode.db")
now = datetime.now(timezone.utc).isoformat()
db.execute("""
    INSERT INTO blackboards (id, project_id, task_description, status, file_path, created_at, updated_at)
    VALUES (?, ?, ?, 'deliberating', ?, ?, ?)
""", [str(uuid.uuid4()), project_id, task_description, file_path, now, now])
db.commit()
db.close()
EOF
```

### Galaxy / Web Frontend (read only)

**Confirmed (2026-06-25):** The galaxy is a **separate Vite dev server process**
(`localhost:3000`) — not an embedded webview. It communicates with the OpenCode
TUI (`localhost:4096`) via HTTP REST and SSE. It has no direct browser access to
the filesystem.

**The access pattern is the Vite `bypass()` proxy** — already used for `/__memory`
(reads `memory.jsonl`) and `/__agents` (reads agent `.md` files). The same pattern
is extended to `/__db` for `opencode.db`. `better-sqlite3` runs in the **Vite/Node
process** (has filesystem access), not in the browser.

```typescript
// vite.config.ts — add /__db alongside existing /__memory and /__agents
import Database from "better-sqlite3";

"/__db": {
  target: "http://localhost:3000",
  bypass(_req, res) {
    const db = new Database(
      "/Users/QTE2362/.local/share/opencode/opencode.db",
      { readonly: true }
    );
    try {
      const projects  = db.prepare("SELECT * FROM projects WHERE status = 'active'").all();
      const boards    = db.prepare("SELECT * FROM blackboards").all();
      const decisions = db.prepare("SELECT * FROM decisions").all();
      const conflicts = db.prepare("SELECT * FROM conflicts").all();
      res.setHeader("Content-Type", "application/json");
      res.end(JSON.stringify({ projects, boards, decisions, conflicts }));
    } finally {
      db.close();
    }
    return false;
  },
},
```

```typescript
// web/src/lib/db-reader.ts — browser-side fetch (same pattern as fetchMemoryGraph)
export async function fetchProjectsGraph(): Promise<ProjectsGraphData> {
  const resp = await fetch("/__db");
  const data = await resp.json();
  return toForceGraphNodes(data);
}
```

**No Express/FastAPI bridge needed.** The existing Vite bypass hook is the bridge.

**Tauri Phase 4 note:** When the Vite dev server is replaced by Tauri (already in
the `feat/web-frontend` roadmap), `better-sqlite3` is swapped for `tauri-plugin-sql`.
The browser-side `fetch("/__db")` calls become `invoke("db_query", {...})` Tauri
commands. Pattern is identical; only the runtime changes.

```
          │
          ▼

  → secretary updates blackboard status: deliberating → awaiting-approval
          │
          ▼
user approves → worker executes → secretary records result
  → blackboard status: done
  → project progress: 4/5 tasks complete
```

## Galaxy Visual Representation

Projects in `opencode.db` appear in the memory galaxy as a distinct node cluster.
See `ARCHITECTURE.md` for the full visual encoding, but the key mappings are:

| DB concept | Galaxy node | Visual |
|---|---|---|
| `projects` row | Project node | Large BMW blue sphere with ring, high mass |
| `blackboards` row | Blackboard node | Medium octahedron, colour = status |
| `decisions` row | Decision node | Small purple diamond |
| `conflicts` row (unresolved) | Conflict node | Small orange/red spiky |
| Agent names in `sections` | Edge to agent node | Directed edge: agent → blackboard |
| `decisions.influenced` | Cross-task edge | Decision node → influenced blackboard |

**The cross-task decision edge is the most valuable visual feature.** When a
decision made in task-003 is still shaping task-008, you see a visible arc between
them — invisible in any list-based PM tool.

### Status colour encoding (blackboard nodes)

| Status | Colour | Animation |
|---|---|---|
| `deliberating` | Yellow | Slow pulse |
| `awaiting-approval` | Amber | Steady glow |
| `executing` | Bright white/blue | Fast pulse |
| `blocked` | Dim red | Slight red tint, no pulse |
| `done` | Faded green | Solid, no animation |

## Secretary Queries (Reference)

Common queries the secretary will run:

```sql
-- Find all active projects
SELECT id, name, status, created_at FROM projects WHERE status = 'active';

-- Get blackboard status summary for a project
SELECT b.id, b.task_description, b.status, b.updated_at
FROM blackboards b WHERE b.project_id = ? ORDER BY b.created_at;

-- Get all decisions for a project (for cross-task enforcement)
SELECT d.decision, d.rationale, d.made_by, b.task_description
FROM decisions d
JOIN blackboards b ON d.blackboard_id = b.id
WHERE b.project_id = ?
ORDER BY d.timestamp;

-- Find unresolved conflicts on active blackboards
SELECT c.*, b.task_description
FROM conflicts c
JOIN blackboards b ON c.blackboard_id = b.id
WHERE c.resolved = 0 AND b.status NOT IN ('done', 'archived');

-- Which decisions influenced a specific blackboard?
SELECT d.decision, d.rationale, d.made_by
FROM decisions d
WHERE json_extract(d.influenced, '$') LIKE '%' || ? || '%';
```

## Migration Strategy

Schema is applied on first secretary activation in Phase 3 via:

```python
# secretary runs this once on startup if tables don't exist
db.executescript(SCHEMA_SQL)
db.commit()
```

The `IF NOT EXISTS` guards make this idempotent — safe to run on every secretary
invocation. No destructive migrations in Phase 3 — tables are additive only.
